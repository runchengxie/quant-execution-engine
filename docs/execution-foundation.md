# 核心执行底座

本文档旨在阐明本项目的核心执行逻辑、系统稳定性的关键前提以及设计边界。

如需了解具体的命令使用方式，请参阅 [命令行参考（cli.md）](cli.md)。  
如需了解项目的文件结构，请参阅 [架构设计（architecture.md）](architecture.md)。  
如需了解测试的运行方式，请参阅 [测试指南（testing.md）](testing.md)。

## 1. 本文档解答什么问题

本文档主要解答以下四个关键问题：

- 本项目在完整量化交易链路中的具体职责范围是什么。
- 订单生命周期中包含哪些核心对象。
- 本地状态、对账机制、风控拦截以及紧急停单机制分别承担何种角色。
- 为什么本项目刻意避免实现某些“大而全的平台化功能”。

## 2. 专注的执行边界

本项目是一个纯粹的执行引擎，仅专注于交易执行层面的核心工作：

- 券商接口的适配与能力矩阵的维护。
- 账户资产快照与市场行情的读取。
- 基于 `targets.json` 目标文件驱动的调仓计划生成、订单提交与状态对账。
- 本地已跟踪订单生命周期的维护。
- 订单执行前及执行过程中的风控检查。
- 提供面向操作员（人工监督）的异常恢复与本地状态维护工具。
- 生成审计日志及用于验证的冒烟测试脚手架。

本项目不负责以下范畴的工作：

- 策略因子的研究与历史数据回测。
- 原始基础行情的清洗导入与特征工程。
- 跨券商的多账户统一路由与资金调拨中台。
- 提供复杂的算法交易调度框架（例如 TWAP 或 POV）。

## 3. 核心对象定义

在当前的系统设计中，订单执行链路被拆分为以下几个核心层级：

1. `OrderIntent`（交易意图）  
   表达“想买卖什么标的、为什么下这笔订单”，以及该意图对应的目标持仓来源、数据截止时间等元数据。
2. `ParentOrder`（母订单）  
   表达某一次交易意图的总体进度状态，例如系统计划请求的总数量、已成交数量、剩余未成交数量，以及当前的汇总状态。
3. `ChildOrder`（子订单）  
   表达向券商发起的某一次具体订单提交尝试（重试或改价时会生成新的子订单）。
4. `BrokerOrderRecord`（券商订单记录）  
   表达券商接口返回的实际订单记录与券商侧的真实状态。
5. `ExecutionFillEvent`（成交事件）  
   表达已确认的局部或全部成交明细记录。

之所以进行这样的拆分，是为了解决实际执行中的以下痛点：

- 在向券商发起提交（`submit`）前，先在本地落盘交易意图，以防止系统异常导致的重复下单。
- 在发生订单重试、改价重提或针对剩余未成交部分继续执行时，能够完整保留所有的历史提交尝试记录。
- 在系统或进程重启后，引擎能够准确识别哪些订单仍在生命周期内，需要继续跟踪。
- 当对账程序拉取到券商状态和成交回报时，能够清晰地将其映射回本地的生命周期模型中。

## 4. 为什么本地状态是核心基础设施

本地持久化的订单状态文件（`outputs/state/*.json`）是整个执行引擎的基石。

它主要承担以下四项核心职责：

- 幂等性保障：防止同一个交易意图被重复提交给券商。
- 异常恢复：确保进程重启后，系统依然能够找回并接管所有正在跟踪的订单。
- 日常运维：所有的查询和干预命令（如 `orders`、`exceptions`、`order`、`retry`、`cancel-rest`）均依赖于本地状态作为数据源。
- 状态对账：当券商侧的状态和成交回报返回后，本地状态是用于合并和更新事实的基准表。

正因如此，本项目优先完善了针对本地订单状态的诊断（doctor）、清理（prune）和修复（repair）工具。

## 5. 对账（`reconcile`）的角色

对账机制负责将券商端的实际执行状态同步至本地系统：

- 刷新本地已跟踪的未完成订单。
- 刷新本地已跟踪的已完成订单。
- 尝试通过比对回补缺失的成交记录。
- 产出状态变更摘要，明确告知操作员本次对账到底有哪些订单发生了实质性变化。

简而言之，对账是连接本地预期状态与券商实际执行状态的修正桥梁。

## 6. 风控与紧急停单机制

本项目侧重于订单执行阶段的风控管理。当前风控主要关注以下指标：

- 单笔订单最大数量限制。
- 单笔订单最大名义金额限制。
- 买卖价差（Spread）限制。
- 市场参与率（Participation Rate）限制。
- 市场冲击（Market Impact）估算。

每一个风控检查项都会输出结构化的决策结果：

- 通过（`PASS`）
- 拦截（`BLOCK`）
- 跳过（`BYPASS`）

如果缺少必要的市场数据（例如无有效报价或缺失日成交量），系统会如实记录为跳过（`BYPASS`），而绝不会强行捏造伪指标来阻断交易。

紧急停单（Kill Switch）机制同样遵循纯粹的执行语义：

- 它仅拦截新的订单提交请求（`submit`）。
- 它不拦截查询、撤单、对账或诊断修复等操作。

这种区分至关重要，因为操作员在紧急阻断新订单生成的同时，仍需要继续进行问题排查和状态恢复。

## 7. 操作员异常恢复链路

系统目前提供了一条清晰的、依赖人工介入的异常接管及恢复路径：

- 状态查询：通过 `orders`、`exceptions`、`order` 命令查看当前执行情况。
- 状态同步：通过 `reconcile` 命令拉取最新的券商状态。
- 订单处置：通过 `cancel`（撤单）、`cancel-all`（全部撤单）、`retry`（重试失败订单）、`reprice`（改价重提）等命令进行干预。
- 部分成交处理：针对部分成交订单，提供 `cancel-rest`（撤销剩余部分）、`resume-remaining`（继续执行剩余部分）以及 `accept-partial`（接受部分成交结果，本地终结预期）等专项操作。
- 维护本地事实：通过 `state-doctor`、`state-prune`、`state-repair` 维护本地状态文件的一致性。

当前的所有恢复链路均被设计为由操作员监督执行的模式。

## 8. 预检（`preflight`）和冒烟测试脚手架的角色

预检（`preflight`）是正式的命令行功能，用于在不改变券商实际状态的前提下，验证账户、网络及数据的就绪情况。

`smoke_operator_harness.py` 则是用于固化流程的冒烟测试脚手架，便于开发者或操作员重复验证以下环节：

- 配置加载（`config`）
- 账户查询（`account`）
- 行情获取（`quote`）
- 调仓执行（`rebalance`）
- 本地订单追踪
- 状态对账（`reconcile`）
- 异常队列检查（`exceptions`）
- 执行后的可选收尾清理工作（`cancel-all`）

在必要时，该脚本可将本次冒烟测试的过程留存为证据 JSON 文件，作为券商侧接入验证的备案。

针对不同券商的定位：
- 长桥（LongPort）：`longport-paper` 目前已适合作为券商侧模拟盘的验证证据。由于实盘资金安全要求较高，长桥实盘仍必须按照单独的“人工监督冒烟操作手册”推进，并且严格要求实盘密钥保存在仓库外部的私有环境中。
- 盈透（IBKR）：`ibkr-paper` 目前作为依赖本地 IB Gateway（基于 TWS API）的模拟盘后端。该后端已打通最小代码执行闭环，但目前留存的仍是“无订单提交（no-order）”级别的证据。下一步的重点是在具备有效行情的环境下，补齐该后端的订单提交、查询、撤销与成交（submit / query / cancel / fill）的冒烟验证证据。

## 9. 为什么模拟盘与实盘的配置隔离也是底座设计

鉴于长桥（LongPort）模拟盘与实盘的验证成熟度不同，系统为两者分别设置了独立的默认配置读取入口。

系统在设计上作出了如下严格约束：
- 模拟盘（`longport-paper`）默认优先读取仓库本地的 `.env` 或 `.env.local` 文件。
- 实盘（`longport`）默认优先读取用户级别的私有配置文件 `~/.config/qexec/longport-live.env`。
- 执行实盘的 `--execute` 动作时，不仅必须设置环境变量 `QEXEC_ENABLE_LIVE=1`，系统还会强制扫描并拒绝在项目根目录本地配置文件（`.env*` / `.envrc*`）中存放实盘密钥的行为。

这样设计的初衷有三点：
1. 保证模拟盘的冒烟测试可以安全、低成本、免干扰地被重复执行。
2. 确保实盘鉴权密钥绝对不会遗留或误提交在项目代码仓库中。
3. 使操作员能够通过 `qexec config` 命令，一目了然地确认当前加载的配置来源到底是本地测试配置还是私有实盘配置。

## 10. 当前设计的取舍

本系统在设计上的取舍非常明确：

- 优先打通交易执行的完整闭环。
- 优先完善操作员的异常恢复链路。
- 优先提升本地状态的一致性与可诊断性。
- 接受并包容不同券商在接口支持度和功能完成度上的差异。

因此，系统目前坚持保留：
- 单账户的执行语义。
- 基于本地追踪状态的订单视图（而非券商全量历史订单簿）。
- 保守的改价策略（采用“先撤单，后按新价格重新提交”的方式）。
- 基于本地 JSON 文件的状态持久化存储。

同时，系统刻意暂缓或不打算实现以下特性：
- 跨券商的多账户统一智能路由。
- 拉取展示券商的完整历史订单簿。
- 基于推送的事件流架构（Event Streaming）。
- 大而全的可视化操作员控制台/仪表板。
- 复杂的订单拆分与算法执行框架。
