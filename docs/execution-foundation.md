# 订单生命周期与执行底座

本页说明订单状态、对账、风控和人工恢复。包结构见 [architecture.md](architecture.md)，命令见 [cli.md](cli.md)。

## 生命周期对象

执行链路把一次交易拆成五类记录：

1. 交易意图记录目标来源、标的、方向和计划数量。
2. 母订单汇总总数量、成交数量、剩余数量和当前状态。
3. 子订单记录每次向券商提交的尝试。重试和改价会产生新子订单。
4. 券商订单记录后端返回的订单编号和状态。
5. 成交事件记录已确认的成交数量、价格和时间。

这组对象可以保留多次提交历史，并把券商回报映射到同一交易意图。

新代码使用的不可变领域对象和 v2 序列化规则见 [typed-execution-domain.md](typed-execution-domain.md)。CLI 与恢复链路当前仍使用 `models.py` 和 `execution_state.py` 中的文件状态 DTO。

## 执行顺序

```text
读取 targets
  → 获取账户和行情
  → 生成调仓计划
  → 执行风控
  → 本地登记交易意图
  → 券商提交
  → 查询与对账
  → 证据留存
```

`qexec rebalance` 默认停在计划阶段。`--execute` 会继续执行后端能力、实盘保护和风险检查。

## 本地状态

本地状态承担以下职责：

- 在提交前登记交易意图，降低重复报单风险
- 在进程重启后找回仍需跟踪的订单
- 为 `orders`、`exceptions` 和恢复命令提供数据源
- 接收券商状态与成交回报的合并结果

本地状态只覆盖本引擎已追踪的订单。券商历史查询提供额外复查视图，不能替代本地生命周期状态。

## 对账

对账会查询后端可提供的订单和成交事实，并更新已追踪记录。输出包括状态变化、成交回补和后端告警。

后端缺少历史查询能力时，对账会按能力矩阵快速失败或返回受限结果。各后端证据范围见 [current-capabilities.md](current-capabilities.md)。

## 风控和紧急停单

当前执行风控包括：

- 单笔数量上限
- 单笔名义金额上限
- 买卖价差上限
- 市场参与率上限
- 市场冲击估算

每项检查产生 `PASS`、`BLOCK` 或 `BYPASS`。市场数据不足时记录 `BYPASS` 和原因，供操作员复核。

紧急停单只拦截新的订单提交。查询、撤单、对账和诊断仍可运行，便于操作员收敛风险并恢复状态。

## 人工恢复

常用恢复顺序：

1. 用 `orders`、`order` 或 `exceptions` 查看本地状态。
2. 用 `reconcile` 拉取最新券商事实。
3. 根据订单状态选择撤单、重试、改价或部分成交处理。
4. 用 `state-doctor` 检查本地文件一致性。
5. 再次对账并保存证据。

`retry` 只处理零成交且已进入终态的订单。部分成交使用 `cancel-rest`、`resume-remaining` 或 `accept-partial`。详细参数见 [cli.md](cli.md)。

## 预检和演练

`preflight` 在不提交订单的前提下检查配置、账户、网络和行情。`health` 汇总预检和本地状态体检。

`project_tools/smoke_operator_harness.py` 用于受监督演练，可覆盖配置、账户、行情、调仓、追踪、对账和收尾清理。券商演练步骤见 [testing.md](testing.md) 中的链接。

## 证据

每次运行可以留下三类材料：

- `outputs/orders/*.jsonl` 中的调仓审计日志
- `outputs/state/*.json` 中的本地执行状态
- `outputs/evidence*` 中的演练证据和复查包

证据包会记录包含、缺失和跳过的材料。凭证文件会被排除。成熟度判断需要同时查看运行环境、操作员步骤和券商回报。

## 配置隔离

长桥模拟盘可以使用项目本地测试配置。长桥实盘凭证保存在用户私有目录或当前进程环境中，项目本地凭证文件会触发保护检查。实盘提交还需要 `QEXEC_ENABLE_LIVE=1`。

完整配置规则见 [configuration.md](configuration.md)。
