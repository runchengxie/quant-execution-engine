# 执行底座

这份文档用于解释这个仓库的执行语义、关键不变量和设计边界。

如果你想知道命令怎么敲，看 [cli.md](cli.md)。  
如果你想知道文件在哪，看 [architecture.md](architecture.md)。  
如果你想知道测试怎么跑，看 [testing.md](testing.md)。

## 1. 这份文档回答什么

它主要回答四件事：

- 这个仓库到底负责执行链路的哪一段
- 订单生命周期里有哪些核心对象
- 本地状态、对账、风控、紧急停单各自扮演什么角色
- 为什么这个仓库刻意不做某些“看起来很像平台”的东西

## 2. 纯执行边界

这个仓库只负责执行域：

- 券商适配层与能力清单
- 账户快照与行情读取
- `targets.json` 驱动的调仓 / 提交订单 / 对账
- 已跟踪订单生命周期
- 订单执行的风控检查
- 人工监督下的恢复与本地状态维护
- 审计日志与冒烟测试工装

这个仓库不负责：

- 因子研究和回测
- 数据导入与特征工程
- 多账户统一路由中台
- 完整算法执行框架，例如 TWAP / POV

## 3. 核心对象

执行链路当前拆成这几层：

1. `OrderIntent`  
   表达“想买卖什么、为什么下这笔单、它来自哪个目标来源 / 截止时点 / 输入来源”。
2. `ParentOrder`  
   表达一次执行意图的总体状态，例如请求数量、已成交数量、剩余数量和当前汇总状态。
3. `ChildOrder`  
   表达一次具体的券商订单提交尝试。
4. `BrokerOrderRecord`  
   表达券商返回的订单记录和券商侧状态。
5. `ExecutionFillEvent`  
   表达已知成交事件。

这样拆分，是为了回答几个真实问题：

- `submit` 前先落 `intent`，如何防止重复下单
- 重试、改价重提、恢复剩余量执行时，如何保留提交尝试历史
- 系统重启后，如何知道哪些订单还要继续跟踪
- 对账返回的状态和已成交记录，如何映射回本地生命周期

## 4. 为什么本地状态是基础设施

`outputs/state/*.json` 是执行层的基础面。

它承担四个职责：

- 幂等：防止同一执行意图重复提交
- 恢复：进程重启后还能找回可跟踪订单
- 运维：`orders` / `exceptions` / `order` / `retry` / `cancel-rest` 等命令都依赖它
- 对账：券商状态和成交回报返回后，需要一个本地事实表来合并

也因此，这个仓库优先补的是订单状态诊断、清理和修复。

## 5. `reconcile` 的角色

它负责把券商侧事实重新拉回本地：

- 刷新本地已跟踪的未结订单
- 刷新本地已跟踪的已结订单
- 尝试回补缺失成交
- 产出变更摘要，告诉操作员这次到底哪些订单发生了变化

换句话说，对账是本地状态和券商状态之间的修正机制。

## 6. 风控与紧急停单

这个仓库做的是订单执行阶段的风控管理。

当前风控主要关注：

- 单笔数量
- 单笔名义金额
- spread
- participation rate
- market impact

每个门禁都会产出结构化决策：

- `PASS`
- `BLOCK`
- `BYPASS`

如果市场数据不足，就记录 `BYPASS`，不生成伪指标。

紧急停单也遵守纯执行语义：

- 它拦新的 `submit`
- 它不拦 `query` / `reconcile` / `doctor` / `repair`

这条区分很重要，因为操作员需要在停掉新下单的同时继续排障和恢复。

## 7. 操作员恢复链路

这个仓库现在有一条明确的人工接管路径：

- 看状态：`orders` / `exceptions` / `order`
- 刷状态：`reconcile`
- 做处置：`cancel` / `cancel-all` / `retry` / `reprice`
- 处理部分成交：`cancel-rest` / `resume-remaining` / `accept-partial`
- 维护本地事实：`state-doctor` / `state-prune` / `state-repair`

当前链路按人工监督方式设计。

## 8. `preflight` 和冒烟工装的角色

`preflight` 是正式 CLI 能力，用来在不改券商状态的前提下检查就绪性。

`smoke_operator_harness.py` 则是固定流程工装，用来重复验证：

- `config`
- `account`
- `quote`
- `rebalance`
- 已跟踪订单查询
- `reconcile`
- 异常队列
- 可选的收尾清理 / `cancel-all`

必要时还可以把这次冒烟写成证据 JSON，作为券商侧验证记录。

对于 LongPort：

- `longport-paper` 现在已经适合做券商侧模拟盘证据
- `longport-paper` 的操作员失败场景冒烟，优先通过单独操作手册重复
- LongPort 实盘通过单独的人工监督冒烟操作手册补实盘证据，实盘密钥保留在仓库外部私有环境

对于 IBKR：

- `ibkr-paper` 当前是本地 IB Gateway over TWS API 依赖型模拟盘后端
- 它已经接入最小执行代码闭环，但已有证据仍是 no-order evidence
- 下一步证据重点是有效行情下的 paper submit / query / cancel / fill

## 9. 为什么模拟盘 / 实盘配置隔离也是底座设计

LongPort 的模拟盘和实盘成熟度不同，当前分别使用不同的默认配置入口。

当前底座刻意约束成：

- `longport-paper` 默认优先仓库本地 `.env` / `.env.local`
- `longport` 实盘默认优先 `~/.config/qexec/longport-live.env`
- 实盘 `--execute` 还要额外通过 `QEXEC_ENABLE_LIVE` 和仓库本地实盘密钥保护

这样设计有三个目的：

- 模拟盘冒烟尽量可重复
- 实盘密钥不落进仓库本地文件
- 操作员可以直接通过 `qexec config` 看出当前命中的配置来源

## 10. 当前设计取舍

这套底座的取舍很明确：

- 优先把执行闭环跑通
- 优先补操作员恢复链路
- 优先补本地状态可诊断性
- 接受不同券商支持度和完成度不一致

所以它现在会保留：

- 单账户语义
- 已跟踪状态视图
- 保守的撤单后重提
- 基于文件的状态存储

同时刻意不急着做：

- 多账户统一路由
- 券商全量订单簿视图
- 事件流
- 大而全的仪表板
- 完整算法执行框架
