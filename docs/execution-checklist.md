# 项目功能清单

> 本项目的首要目标是跑通最基础的交易执行流程，并支撑必要的技术验证。
> 这份功能清单旨在明确项目的完成标准与功能边界，保持设计的克制。
> 市面上已有 QuantConnect/LEAN、vn.py 等成熟的量化执行框架，本项目无意重复造轮子，更不会追求做一个大而全的平台。

## 状态标记

- `[x]` 已完成，已在代码库中落地
- `[~]` 已有雏形，但仍存在明显限制或验证证据不足
- `[ ]` 计划开发，且符合当前项目边界
- `[-]` 暂缓开发，当前阶段优先级较低
- `[!]` 明确不开发，防止项目过度膨胀

## 当前目标边界

本清单基于以下核心定位制定：

- 纯粹的量化交易执行代码库
- 优先支持单账户运行逻辑
- 侧重于低频或半自动的交易执行场景
- 完全通过标准化的 `targets.json` 驱动券商端的调仓、下单与对账
- 提供允许人工介入的异常接管与运维链路
- 现已支持长桥实盘及模拟盘（`longport-paper`）、Alpaca 模拟盘和盈透模拟盘（`ibkr-paper`）；允许不同券商后端的成熟度与验证依据存在差异。

若未来项目目标演变为研究、回测、实盘一体化平台，或跨券商的多账户统一中台，则本清单将不再适用。

## 完成线定义

当满足以下条件时，即标志着本项目已成功跑通核心执行闭环，且守住了不盲目扩张的边界：

1. 能够稳定完成 `account`（账户查询） -> `quote`（行情获取） -> `rebalance --execute`（执行调仓） -> `reconcile`（状态对账） 的主干流程。
2. 能够在本地执行状态中，对已追踪的订单进行查看、撤销、重试、恢复及人工接管。
3. 能够在券商端返回数据迟到、部分缺失、部分成交，或需要人工介入刷新时，完成基础的状态恢复。
4. 能够清晰界定功能代码已实现与可靠性已获充分验证之间的区别。

## 核心清单

### 1. 最小执行主路径

- `[x]` 仅接受标准格式的 `targets.json` 作为唯一的实盘调仓输入
- `[x]` `qexec config`（查看配置）
- `[x]` `qexec account`（查询账户概览）
- `[x]` `qexec quote`（获取实时行情）
- `[x]` `qexec rebalance`（调仓预演/Dry-run）
- `[x]` 附加 `--execute` 参数执行 `rebalance` 的实盘与模拟盘报单路径
- `[x]` 将调仓审计日志结构化输出至 `outputs/orders/*.jsonl`
- `[x]` 将本地执行状态持久化输出至 `outputs/state/*.json`
- `[x]` 券商适配器能力矩阵声明
- `[x]` 正式的 `qexec preflight`（执行前就绪性检查）
- `[x]` `qexec evidence-maturity`（生成券商代码支持度与验证证据成熟度报告）
- `[x]` `qexec evidence-pack <run-id>`（按单次运行编号打包收集复查证据）

### 2. 订单生命周期与本地状态

- `[x]` 确立交易意图（Intent）、母订单（Parent Order）、子订单（Child Order）与成交事件（Fill Event）的核心数据模型
- `[x]` 幂等提交机制，防止同一交易意图被重复报单
- `[x]` 本地券商订单追踪机制
- `[x]` 触发对账（`reconcile`）时同步刷新本地追踪的未完成订单
- `[x]` 触发对账时同步刷新本地追踪的已完成订单
- `[x]` 将券商成交明细回补至本地状态模型中
- `[x]` 输出对账前后的差异与状态变更视图
- `[x]` 紧急停单（Kill Switch）与基础风控门禁逻辑
- `[x]` 本地状态文件维护工具（`state-doctor` / `state-prune` / `state-repair`）

### 3. 运维与人工接管入口

- `[x]` `qexec orders`（查看本地追踪状态的订单）
- `[x]` `qexec exceptions`（查看需要介入的异常队列）
- `[x]` `qexec order <order-ref>`（查看单笔订单的完整生命周期）
- `[x]` `qexec reconcile`（手动触发与券商后台的状态对账）
- `[x]` `qexec cancel <order-ref>`（撤销指定订单）
- `[x]` `qexec cancel-all`（一键撤销当前账户下所有追踪中的挂单）
- `[x]` `qexec retry <order-ref>`（重试失败或被拒的订单）
- `[x]` `qexec retry-stale --older-than-minutes N`（重试过期未成交的静止挂单）
- `[x]` `qexec cancel-rest <order-ref>`（针对部分成交订单，撤销其剩余未成交数量）
- `[x]` `qexec resume-remaining <order-ref>`（针对部分成交订单，继续发起新的子订单执行剩余数量）
- `[x]` `qexec accept-partial <order-ref>`（在本地接受局部成交结果，完结该订单意图）
- `[x]` 明确原则：以上所有干预命令均仅围绕“本地已追踪订单”和“本地执行状态”展开，不涉及查询或操作券商端的全量历史订单簿。

### 4. 现在已经补实的能力

- `[x]` 券商拒单与告警归一化输出： 当前的命令行摘要、异常队列视图及单笔订单详情，均能将券商下发的原始错误信息转化为统一的诊断代码，并提供下一步操作建议。
- `[x]` 部分成交的人工恢复链路： 现已全面提供 `cancel-rest`、`resume-remaining` 和 `accept-partial` 三条专门的操作员接管路径。
- `[x]` 预检功能转正： 预检（`preflight`）已从内部测试工具升级为正式的终端命令行功能。
- `[x]` 对账视图升级： 状态对账（`reconcile`）的输出已从单纯的数量统计，升级为详尽的状态变更差异视图。
- `[x]` 测试证据留存： 面向操作员的冒烟测试脚手架现已支持可选的测试证据文件导出功能。
- `[x]` 长桥（LongPort）配置隔离与来源可见性： 模拟盘路径默认优先读取代码库本地的 `.env` 文件，而实盘路径则强制优先读取用户私有目录（`~/.config/qexec/longport-live.env`）；通过 `qexec config` 命令可清晰查阅各项核心参数最终的加载来源。
- `[x]` 券商验证成熟度报告： `qexec evidence-maturity` 命令能够清晰呈现各家券商（长桥实盘/模拟盘、Alpaca、盈透）在代码逻辑与自动化证据留存方面的支持状态，并指出存在的缺口及下一步的测试建议。
- `[x]` 审计与证据打包归档： `qexec evidence-pack <run-id>` 能够根据运行编号，自动汇集审计日志、目标持仓、本地状态快照、冒烟测试结果及操作员人工备注，并生成清单文件与复查压缩包。
- `[x]` 风控降级显式提示： 预检环节会明确提示哪些依赖市场数据的风控项可能因有效报价或成交量数据缺失而被降级跳过（BYPASS）；调仓的输出面板与审计日志中也会如实记录风控跳过的具体次数与原因。

### 5. 当前仍值得继续补的功能

- `[x]` 长桥模拟盘（`longport-paper`）接入： 目前已通过模拟盘凭证顺利走通了券商后台的下单、查询、撤单与对账全链路，并成功留存了由人工监督的模拟盘冒烟测试证据链。
- `[~]` 长桥实盘业务闭环的端到端证据仍需夯实：配置加载、执行预检、账户读取与行情获取等只读操作已验证通过；下一步必须补充最小实盘环境下的 `rebalance --execute` 验证证据。该通道的最终成熟度将以成熟度报告及本地留存的打包证据为准。
- `[~]` 盈透模拟盘（`ibkr-paper`）接入已落地，但真实订单证据链待补齐： 当前采用依赖本地IB Gateway 配合 TWS API的运行路线，已支持美股正股的最小闭环。已有无报单（No-order）的测试记录证明了网关连通性及账户/对账链路的可用性；但仍需在具有有效行情的环境下，通过下一次冒烟测试补齐真实的报单、成交与撤单验证证据。
- `[~]` 扩展失败场景的回归测试： 目前已涵盖了实盘行情缺失跳过、脚手架运行拦截以及部分成交后的状态恢复测试，但仍需进一步覆盖更多真实的失败场景（如网络超时、区域限制引发的拒单等）。
- `[~]` 细化券商特定拒单原因分类： 现有的统一诊断层已具备基础归类能力，但对各家券商底层原始错误码的细粒度解析仍有优化空间。

### 6. 有骨架，但还没到“完成”的项目

- `[~]` 多券商支持： 长桥实盘及模拟盘、Alpaca 模拟盘、盈透模拟盘均已建立基础底座；由于开发侧重点不同，不强求各家券商达到完全一致的成熟度。
- `[~]` 子订单尝试与调度管理： 当前已支持手工重试、改价及继续执行剩余订单；但更完善的全自动订单调度机制（Scheduler）暂未展开。
- `[~]` 本地状态持久化增强： 现有的 JSON 文本存储机制对单机、低频和手工盯盘场景已足够应对；更强壮的持久化存储层（如引入关系型数据库）暂未展开。

## 暂缓项

- `[-]` 盈透（IBKR）的实盘接入，以及更广泛的资产类别与复杂订单类型的支持。
- `[-]` 引入 SQLite 等数据库作为底层状态存储。
- `[-]` 基于事件流（Event Streaming）架构的券商实时数据推送响应机制。
- `[-]` 更细粒度的业务监控指标图表与告警通知推送。

当前阶段的绝对优先级是夯实既有执行闭环的复查证据链与异常失败场景的处理机制。

## 明确不做

- `[!]` 将策略研究、回测系统、Alpha 信号挖掘或历史数据导入等模块回流至本执行仓库中。
- `[!]` 构建跨越多个券商底层的真实多账户统一智能路由中台。
- `[!]` 在项目初期就强行统一所有券商的订单类型（Order Type）、有效时间（Time In Force）及交易时段（Session）的底层细节语义。
- `[!]` 引入 TWAP、POV 等复杂的算法订单拆解与执行调度框架。
- `[!]` 开发大而全的可视化仪表盘或面向操作员的 Web 控制台界面。
- `[!]` 强求所有接入的券商必须在同一时间达到“最高成熟度”的一等公民标准。

## 当前建议顺序

1. 优先补齐长桥实盘链路的复查证据： 只读验证已确认无误，下一步的重点是安全地推进最小规模的实盘冒烟测试并妥善留存档案。
2. 补充盈透模拟盘的真实订单流转证据： 必须在具有有效市场行情报价的环境下，补齐下单、查询、撤单与成交环节的模拟盘测试依据。
3. 扩充失败场景的自动化回归测试： 优先覆盖券商明确拒单、迟到成交记录、撤单处于等待状态（`pending-cancel`），以及因行情、可用区与网络异常导致的阻断场景。
4. 贯彻执行审计打包流程： 针对每一次关键的冒烟测试，务必执行 `qexec evidence-pack <run-id>` 命令，将审计日志、本地状态、目标持仓输入、冒烟测试 JSON 文件及人工复查备注统一归档打包为复查凭证。
5. 根据实际的运维需求，继续细化特定券商的拒单分类与诊断提示语。
6. 继续将 Alpaca 模拟盘作为低成本、高稳定性的日常回归与冒烟测试基准环境。

## 维护原则

1. 新增功能只有在能直接提升或闭环交易执行能力时，才能被列入 `[x]`（已完成）或 `[ ]`（计划中）的范畴内。
2. 若某项功能设计明显带有平台化或大一统过度抽象的倾向，应优先将其划入 `[-]`（暂缓）或 `[!]`（明确不做）的类别中。
